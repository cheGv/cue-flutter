# Cue audit — 8 May 2026

**Mode:** Systematic surface auditing. No fixes during the audit.
Audit produces findings; findings get triaged into phase tickets;
fixes happen after.

**Method:** 11 surfaces × 5 questions. Answer each. Mark verified
vs. assumed vs. untested.

## The 5 questions for every surface

1. **Persistence** — Does it persist what it claims to persist?
   Verify in the DB, not just the UI.
2. **Entry points** — Does every entry point reach it cleanly?
   List them all. A surface with N entry points has N regression
   paths.
3. **Empty / loading / error states** — Are they handled
   explicitly, or does the surface fall through to whatever
   Flutter renders by default?
4. **Back-button + browser-refresh** — Is state lost? Does the
   route resolve? Does the user end up where they expect?
5. **Chrome consistency** — Does it match Phase 4.0 register
   (paper cream, navy primary, amber accent, hairline borders,
   eyebrow labels)? Anything in legacy teal / Material green /
   Phase 2 register gets logged but NOT fixed during the audit.

## After the audit

Triage findings into:
- **Quick wins** (single-line bugs, <10 min each) → Phase 4.0.7.x
  hotfixes
- **Structural items** (missing patterns, shared widgets) →
  dedicated phase tickets
- **Open product questions** (where right behavior isn't clear)
  → separate conversation, not a code commit

Update `docs/audit-theme-tracker.md` with any new themes
discovered.
